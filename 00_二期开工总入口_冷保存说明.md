1. 当前已经冷保存了什么

4. 遇到不确定细节时按什么顺序裁决
5. 哪些工程要求从第一天起就不能丢

如果后续接手者时间很紧，至少应先读完本页，再去读执行手册和总蓝图。

## 3. 今晚正式开工时的唯一推荐顺序

今晚如果开始做，推荐严格按下面的顺序推进：

1. 先做项目基线冻结
2. 先做最小实验底座
3. 先做最小观测底座
4. 先把 BAT / 启动器 / 中文操作路径留好
5. 再做文本输入最小闭环

展开成更具体的待办，就是：

1. 冻结目录结构
2. 冻结配置加载与 schema 基线
3. 冻结 run manifest 结构
4. 实现 JSONL 追加写日志底座
5. 实现最小 live 状态 API
6. 实现最小 tick 回放 API
7. 约定观测台启动 BAT / 单次实验 BAT / 批量实验 BAT 的路径与命名
8. 再进入文本输入最小闭环

今晚明确不建议优先做的事情：

1. 视觉感受器
2. 听觉感受器
3. Agent sandbox
4. 遗忘系统
5. 复杂前端炫技展示
6. 过重的自动调参器

当前已完成的最小落地成果：

1. `observatory_v2/` 最小服务骨架
2. 配置文件与轻量 schema 校验
3. `run_manifest/v1`

5. `tick_metrics/v1`

## 4. 遇到不确定细节时，必须这样裁决

这是今晚最重要的口径之一，禁止边实现边漂移：

### 第一优先级：二期已批准草案

优先查：

1. `AP二期_工程落地顺序规划_冷保存执行手册.md`
2. `AP二期_V2总蓝图_详细设计草案.md`
3. `AP二期_HDB-V2_详细设计草案.md`
4. `AP二期_观测台V2_详细设计草案.md`
5. `AP二期_感受器V1_设计草案.md`
6. `AP二期_先天规则编码_底层可追踪条件总表与建议规则表.md`

### 第二优先级：V1 已验证实现

如果二期文档还没细到实现级，则参考 V1 中已经被证明有效的：

1. 运行骨架
2. run / manifest / metrics / progress 方案
3. 观测台已有的实验管理和输出路径
4. 已验证的操作便利性设计

但迁移时必须明确记录：

1. 保留了什么
2. 修改了什么
3. 为什么修改

### 第三优先级：论文与一贯哲学

如果二期文档和 V1 都没有直接答案，再参考：

1. 论文中的理论表达
2. 一贯哲学
   - 白箱优先
   - 长期闭环优先
   - 不为了 demo 牺牲结构正确性
   - 优先让主线收敛，而不是增加花哨机制

### 第四优先级：最小复杂度优秀方案

如果仍然没有直接答案，就选：

1. 更简单
2. 更稳定
3. 更可观测
4. 更易测试
5. 更易导入导出

## 5. 这次落地必须从第一天就保留的工程品质

以下要求不是后补要求，而是正式工程要求：

1. 中文优先的用户可见文案
2. 关键日志和 API 字段稳定可读
3. 观测便利性从最小版本就开始保留
4. BAT 一键启动路径要尽早留出
5. 长跑内存与输出规模要从架构上约束
6. 主机制必须有对应的白箱证据

尤其要注意：

1. 不要再走 V1 那种“巨型运行态报告 + 前端频繁整包轮询”的重路径
2. 观测台应坚持“在线轻量，离线重建”
3. 运行日志应优先采用 append-only 的 JSONL 形式
4. 详细细节更多依赖离线 inspector / replay / 物化视图，而不是主循环里长期堆大对象

## 6. 今晚实施时的最小成功标准

如果今晚只做 Phase 0 / Phase 1，那么最小成功标准应是：

1. 项目目录骨架明确
2. 配置和 schema 有基础加载能力
3. run manifest 可以稳定生成
4. JSONL 日志可以稳定追加写
5. live API 可查看当前运行摘要
6. tick replay API 可查看单 tick 摘要
7. 启动路径有统一约定
8. 后续能自然接上文本输入最小闭环

只要这几项打通，今晚就算把“后续不迷路的工程底座”立住了。
