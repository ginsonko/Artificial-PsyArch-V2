## 1. 本手册的定位

这份文档不是泛泛的“任务列表”，而是 **AP 二期正式落地时的执行手册**。

它的目标不是再讨论方向，而是确保：

1. 开工时不乱序
2. 每个阶段先把设计细节钉死
3. 设计没有钉死前，不轻易编码
4. 编码后一定有可观测证据与验收测试
5. 每通过一阶段，再进入下一阶段
6. 后续无论是人类开发者还是大模型接手，都能按同一套口径继续推进

换句话说，这份文档要解决的是：

> **如何把现在已经形成的二期蓝图，变成一个真正可执行、可收敛、可验收、可回滚、可冷保存延续的工程实施流程。**

## 2. 核心执行原则

AP 二期的实施必须遵守以下原则。

### 原则 1：阶段化推进，不跳步

不允许因为“某个后续模块很诱人”就跳过前面的基础阶段。  
特别是：

1. 不允许在观测底座没稳定时先猛做多模态
2. 不允许在文本最小闭环没跑稳时先猛做 Agent
3. 不允许在导入导出协议没清楚时先堆 checkpoint

### 原则 2：先设计冻结，再编码

每个阶段都必须先明确：

1. 术语
2. 对象
3. 输入输出
4. 数据结构
5. 不变量
6. 日志字段
7. 失败模式
8. 验收标准

只要这些没写清楚，就不应进入大规模实现。

### 原则 3：每阶段都必须可观测

如果一个阶段做完了，但观测台或日志看不见它的关键行为，则该阶段视为未完成。  
因为 AP 的核心优势之一就是白箱。

### 原则 4：每阶段都必须有验收

验收不是“看起来能跑”。  
至少应包括：

1. 单元测试
2. 集成测试
3. 观测验证
4. 长跑稳定性验证
5. 回归验证

### 原则 5：先文本稳定，再多模态，再 Agent

二期的最稳实施路线是：

1. 先打通文本版 V2 核心大脑
2. 再接视觉 / 听觉身体
3. 再接电脑控制 / Agent 行动

### 原则 6：不确定细节优先参考已批准口径

如果某个设计细节在实现时不确定，优先级应固定，不能边做边乱改。  
详见第 3 节。

### 原则 7：观测便利性不是附属品

一键启动、中文显示、日志可读、实验可回放、run 可管理，这些都不是“以后再补”的 UI 小事，  
而是二期平台能否持续工作的基础。

## 3. 决策优先级与设计歧义处理规则

这是开工时最容易漂移的地方，因此必须写死。

当某个设计细节不确定时，统一按以下优先级决策：

### 优先级 1：当前已批准的二期草案

优先参考：

1. [AP二期_V2总蓝图_详细设计草案.md](H:\AP原型实验第二期\AP二期_V2总蓝图_详细设计草案.md)
2. [AP二期_HDB-V2_详细设计草案.md](H:\AP原型实验第二期\AP二期_HDB-V2_详细设计草案.md)
3. [AP二期_感受器V1_设计草案.md](H:\AP原型实验第二期\AP二期_感受器V1_设计草案.md)
4. [AP二期_先天规则编码_底层可追踪条件总表与建议规则表.md](H:\AP原型实验第二期\AP二期_先天规则编码_底层可追踪条件总表与建议规则表.md)
5. [AP二期_观测台V2_详细设计草案.md](H:\AP原型实验第二期\AP二期_观测台V2_详细设计草案.md)

这些文档是当前最高优先级工程口径。

### 优先级 2：V1 中已经被验证有效的细节实现

如果二期文档未细到实现级，而 V1 某处已经有：

1. 明确的运行逻辑
2. 已验证的观测逻辑
3. 已验证的操作便利性

则允许优先复用或迁移，但要求：

1. 口径要与 V2 不冲突
2. 迁移时要显式记录“保留 / 修改 / 废弃”的理由

### 优先级 3：论文与一贯哲学

如果 V2 文档和 V1 实现都没有明确答案，则参考：

1. 论文中对 AP 的理论阐述
2. 一贯哲学
   - 白箱优先
   - 长期闭环优先
   - 不为了短期 demo 牺牲结构正确性
   - 优先让主线收敛，而不是加炫技机制

### 优先级 4：最小复杂度的优秀设计

若以上都无法直接回答，则默认采用：

1. 更简单
2. 更稳定
3. 更可观测
4. 更易测试
5. 更易导入导出

的方案。

### 必须避免的歧义处理方式

不允许：

1. 因为赶进度随便选一版
2. 因为实现麻烦就偷偷改理论口径
3. 因为观测不好做就把日志省掉
4. 因为多模态复杂就先做黑箱 embedding 顶上

## 6. 每阶段统一工作法

每个阶段都必须按同一套工作法推进。

### 6.1 设计冻结

至少写清楚：

1. 模块职责
2. 输入输出
3. 数据结构
4. 不变量
5. 失败模式
6. 验收标准

### 6.2 观测字段冻结

在实现前，先确定：

1. 哪些指标必须记录
2. 哪些对象 top 必须记录
3. 哪些阶段 timing 必须记录
4. 哪些异常要报警

### 6.3 最小夹具设计

在实现前，先准备：

1. 最小正例
2. 最小反例
3. 边界输入
4. 长跑样例

### 6.4 最小实现

先做：

1. 正确
2. 可观测
3. 可测试

再做优化，不要反过来。

### 6.5 验收测试

每阶段必须至少有：

1. 单元测试
2. 集成测试
3. 观测验证
4. 稳定性测试
5. 回归测试

### 6.6 冻结结果

只有当：

1. 文档已更新
2. schema 已更新
3. 回归集已增加
4. 启动方式可用
5. 观测能看见

才允许进入下一阶段。

### 阶段验收

1. 音频窗进入系统稳定
2. 持续背景音会疲劳压低
3. 听觉对象能参与召回
4. 观测台能看波形与采样窗

## 22. 观测便利性与操作体验硬要求

二期不是实验室里只会命令行的人自己用的系统。  
必须从一开始就对操作体验提出硬要求。

### 22.1 必须具备的一键启动能力

至少要有：

1. 一键启动观测台
2. 一键启动批量实验
3. 一键修复或重建必要缓存 / derived
4. 一键查看最新 run

### 22.2 必须具备的最小观测视图

即使在最早阶段，也应尽量做到能看到：

1. 当前 run 状态
2. 当前 tick
3. 当前主线 top
4. 当前 `R_state`
5. 当前 Bn
6. 当前 `C*`
7. 当前 `A_focus`

### 22.3 禁止纯黑盒推进

如果某阶段完成后：

1. 看不见关键过程
2. 只能看最终输出

则视为阶段未完成。

## 23. 语言、命名、日志与报告要求

### 23.1 用户可见内容默认中文优先

除非有强烈技术原因，否则：

1. 页面文案
2. BAT 名称
3. 报告标题
4. 错误提示
5. 实验摘要

都应优先中文，必要时中英并存。

### 23.2 命名风格

建议：

1. 代码内部保持稳定英文 id
2. 用户可见层提供中文 label
3. 日志层同时保留 id 和 label

### 23.3 日志可读性

日志不是只给程序读，也要给人和后续大模型读。

因此：

1. JSON 字段名稳定
2. 关键枚举清晰
3. 不用模糊缩写堆叠
4. 有摘要层，不全是原始数字

### 23.4 报告导出

每个阶段完成后，应能导出：

1. 当前阶段完成说明
2. 测试结果摘要
3. 已知问题
4. 下一阶段建议

## 24. BAT / 启动器 / 快速运行体验要求

用户已经明确强调这一点，因此必须提前冻结。

### 24.1 启动器原则

1. Windows 下优先提供双击可启动方案
2. 尽量不要求用户先手打一长串命令
3. 启动器默认隐藏无关噪音
4. 启动失败时能给出清晰中文提示

### 24.2 建议的启动器集合

至少建议后续提供：

1. `启动观测台.bat`
2. `启动单次实验.bat`
3. `启动批量实验.bat`
4. `查看最新运行结果.bat`
5. `修复缓存与重建视图.bat`

### 24.3 启动器的最低要求

每个 BAT 或启动器脚本至少应做到：

1. 检查依赖
2. 检查端口占用
3. 输出运行地址
4. 输出日志路径
5. 输出停止方法

### 24.4 与观测台的联动要求

启动观测台后应尽可能：

1. 自动打开本地页面
2. 自动定位最新 run
3. 或至少打印清晰的访问地址和操作提示

## 26. 今晚开工时的推荐起手顺序

如果今晚就开始落地，最推荐的顺序是：

1. 阶段 0 的剩余冻结项全部补齐
2. 先做阶段 1：实验底座与最小观测底座
3. 立刻留出 BAT/启动器路径

### 今晚最值得优先做的具体事情

### 今晚不建议做的事情

## 27. 最终执行口径

把整份手册压缩成一句话，就是：

> **AP 二期必须按“先冻结设计，再做最小实现，再做白箱验收，再冻结结果，再进入下一阶段”的顺序推进；**
>
> **先做文本最小闭环和实验底座，确认 `R_state -> Bn -> C_i -> C* -> A_focus` 主线稳定，再逐步接规则、多模态、导入导出、Agent 与遗忘系统；**
>
> **一切不确定细节，优先参考已批准二期草案，其次参考 V1 已验证细节，再参考论文与一贯哲学；**
>
> **观测便利性、中文化、日志可读性与一键启动体验必须从一开始就被视为正式工程要求，而不是后补 UI。**

这就是 AP 二期当前正式推荐的工程落地顺序规划冷保存执行手册。
