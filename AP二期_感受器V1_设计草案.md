## 1. 文档目的

本文档不是在 HDB-V2 总草案之外另起一个系统。  
它的目标是把“视觉感受器”和“听觉感受器”正式定义成 AP 二期认知闭环的前端输入层。

它要回答五个问题：

1. 感受器到底负责什么，不负责什么
2. 感受器如何把高吞吐原始流压缩成 AP 可承受的固定预算刺激包
3. 视觉和听觉的最小工作单位应该是什么
4. 感受器如何与 HDB-V2、状态池、注意力、行动接口衔接
5. 如何保证这一步不会把 AP 原本的白箱优势变成黑箱前处理

这份文档的默认读者包括：

1. AP 架构发起人
2. 后续实现视觉/听觉模块的人类开发者
3. 需要补充设计、写模块骨架、写测试或做理论审查的大模型

---

## 2. 设计目标

感受器 V1 的目标不是一上来实现完整具身感知，而是做出一个：

1. **可运行**
2. **可控复杂度**
3. **白箱可观察**
4. **能接 HDB-V2**
5. **能为后续具身版本留接口**

的前端输入层。

因此本阶段的目标是：

1. 支持图片 / 截图 / 低帧率视频片段进入视觉感受器
2. 支持音频文件 / 麦克风短窗 / 视频音轨片段进入听觉感受器
3. 每个 cognitive tick 只向主闭环提交固定预算刺激包
4. 支持感受器级疲劳降噪
5. 支持视觉焦点中心与焦点行动竞争
6. 支持感受器白箱观测

---

## 3. 核心结论

感受器 V1 的正式口径可以压缩成以下几句：

1. **像素和原始音频采样点不是 AP 主流程直接使用的 SA。**
2. **感受器必须先把高吞吐原始流压缩成固定预算刺激包，再交给主闭环。**
3. **视觉必须保留稳定坐标和焦点相对坐标两套空间表达。**
4. **听觉一期可以单通道简化，但不应宣称具备真实空间定位能力。**
5. **重复刺激疲劳应上移到感受器层，以避免持续噪音和稳定背景吞噬认知预算。**
6. **感受器时间节拍与认知时间节拍必须解耦。**
7. **感受器前端也必须是白箱的。**

## 4. 与一期 `text_sensor` 的继承关系

AP 二期感受器 V1 不应完全脱离一期现有 `text_sensor` 的设计语言。  
相反，建议显式继承以下口径：

### 4.1 继承的一期优点

一期 `text_sensor` 已经证明了几件重要事情：

1. 感受器是前端输入整理器，而不是语义解释器
2. 感受器可以输出统一 `stimulus_packet`
3. 感受器可以带自己的运行态与疲劳历史
4. 感受器可以把重复刺激逐步抑制
5. 感受器可以输出 `sensor_frame` 供白箱调试

这些都应该被二期继续保留。

### 4.2 二期需要超越的地方

一期 `text_sensor` 主要面对的是离散文本输入。  
二期视觉/听觉感受器面对的是连续流，因此必须新增：

1. `sensor micro-tick`
2. 感受器缓冲
3. 固定预算采样
4. 多因素显著图
5. 焦点中心
6. 更复杂的空间/时间字段

因此二期不是“照搬 text_sensor”，而是：

> **继承它的接口哲学、白箱精神和疲劳思想，但扩展到连续多模态场景。**

## 5. 感受器 V1 的统一职责边界

### 5.1 感受器负责什么

感受器 V1 负责：

1. 接收原始外界输入
2. 在感受器层做基础采样和预算控制
3. 形成基础刺激单元
4. 生成有限的属性实例候选
5. 应用感受器级疲劳抑制
6. 输出统一刺激包
7. 维护感受器运行态快照

### 5.2 感受器不负责什么

感受器 V1 不负责：

1. 长期记忆存储
2. Bn 召回
3. `C_i / C*` 预测包生成
4. 抽象回声
5. 低频结晶
6. 状态池生命周期
7. 注意力最终选峰

换句话说：

> **感受器负责“把世界整理成可进入 AP 的刺激对象集合”，不负责后续认知闭环的主计算。**

## 6. 两层时间：sensor micro-tick 与 cognitive tick

这是感受器 V1 最重要的设计之一。

### 6.1 为什么必须分两层时间

视觉和听觉原始输入的更新速度，往往比认知 tick 快得多：

1. 视频可能是 25-60fps
2. 音频采样率可能是 16kHz 甚至更高
3. AP 主闭环认知 tick 可能只希望是 5-20Hz

若三者混在一起，会造成：

1. 复杂度失控
2. 快速变化丢失
3. 顺序和归因混乱

### 6.2 正式定义

建议定义：

1. **sensor micro-tick**
   感受器自己的采样与缓冲节拍

2. **cognitive tick**
   AP 主闭环的一轮认知计算节拍

感受器在多个 micro-tick 内累计和筛选信号，  
到 cognitive tick 边界时，再统一产出一个固定预算刺激包。

### 6.3 一个例子

假设：

1. 音频每 10ms 进入一个小窗
2. 视觉每 40ms 进入一帧
3. AP 主闭环每 100ms 跑一个 tick

那么在一个认知 tick 中：

1. 听觉可能经历 10 个 micro-tick
2. 视觉可能经历 2-3 个 micro-tick
3. 感受器内部先汇总
4. 主闭环最后只读取一次压缩结果

---

## 7. 固定预算刺激包原则

### 7.1 这是硬约束，不是可选优化

二期感受器 V1 必须满足：

> **每个感受器在每个 cognitive tick 只能向主闭环提交一个固定容量或固定上限容量的刺激包。**

### 7.2 为什么必须这样

因为 AP 主闭环复杂度与输入刺激长度强相关。  
如果感受器原始吞吐量直接灌入主闭环，会造成：

1. SA 竞争成本爆炸
2. Bn 候选规模不稳定
3. `C_i / C*` 和回声叠加被级联放大
4. 实验结果不可比较

### 7.3 固定容量与有效填充量

建议区分：

1. **固定容量**
   例如视觉每 tick 最多 64 个单位，听觉最多 32 个单位

2. **有效填充量**
   当前 tick 实际重要输入不足时，可以少于上限，用空槽填充

这意味着：

1. 主闭环复杂度稳定
2. 无需为了凑满容量强造伪刺激

### 7.4 建议的初始预算

建议从较保守的预算开始：

1. `L_text = 32`
2. `L_visual = 64`
3. `L_audio = 32`

后续再按硬件和热路径测量调参。

## 8. 感受器统一输出口径

建议二期感受器统一对外提供三个核心产物：

1. `sensor_frame`
2. `stimulus_packet`
3. `runtime_snapshot`

### 8.1 `sensor_frame`

`sensor_frame` 是当前感受器这一轮对外部输入的原始整理快照。  
它不等于长期记忆，只是当前感受器产出记录。

建议保留字段：

1. `id`
2. `object_type = sensor_frame`
3. `sub_type`
4. `frame_no`
5. `trace_id`
6. `tick_id`
7. `micro_tick_range`
8. `source`
9. `sampling_summary`
10. `unit_ids`
11. `attribute_instance_ids`
12. `debug`

### 8.2 `stimulus_packet`

这是主闭环真正读取的对象。

建议统一保留：

1. `sa_items`
2. `csa_items`
3. `grouped_sa_sequences`
4. `energy_summary`
5. `sensor_budget_summary`
6. `fatigue_summary`
7. `sampling_summary`

### 8.3 `runtime_snapshot`

供观测台与调试使用，至少包括：

1. 当前预算使用情况
2. 当前疲劳状态统计
3. 当前缓冲区统计
4. 当前焦点状态
5. 最近几轮采样摘要

## 10. 感受器级重复刺激疲劳

### 10.1 为什么必须上移到感受器层

若重复刺激只在状态池或后续层才被抑制，会浪费前端预算。  
对视觉/听觉来说，这尤其危险：

1. 稳定背景会反复占据输入槽位
2. 底噪会持续抢占听觉预算
3. 主闭环会花算力处理本不该反复进入的东西

因此必须在感受器层先做一轮 habituation。

### 10.2 设计原则

重复疲劳不应只按“完全相同底层单元”计算。  
建议按以下粒度之一建 key：

1. `family + 区域 + 粗数值桶`
2. `family + 跟踪对象`
3. `family + 模式簇`

### 10.3 疲劳不应把重要持续刺激彻底压没

重复抑制要结合任务和注意调制：

`ER_ext_final = ER_ext_raw * gain_habituation * gain_task * gain_attention`

这样：

1. 普通背景可逐步降噪
2. 任务相关持续对象仍可维持可见
3. 已被注意对象可保持更高进入率

### 10.4 高效实现建议

参考一期 `text_sensor` 的历史窗口思想，  
但为控制成本，建议优先使用 EMA 近似：

`h_t = lambda * h_(t-1) + (1-lambda) * presence_t`

再映射成抑制增益：

`gain_habituation = max(g_min, 1 - beta * h_t^gamma)`

这样每个疲劳 key 只需常数级状态。

## 11. 视觉感受器 V1

### 11.1 输入范围

视觉感受器 V1 建议支持：

1. 静态图片
2. 屏幕截图
3. 低帧率视频流
4. 后续可扩展到摄像头帧

### 11.2 最小工作单位

视觉主流程不直接使用像素。  
建议采用：

1. 固定 patch
2. 网格 tile
3. 超像素块（后续可选）
4. 运动变化块（视频时）

### 11.3 三路采样结构

推荐每个 cognitive tick 的视觉预算由三部分构成：

1. **全局粗采样**
   - 低分辨率覆盖整幅图
   - 保证系统不会完全忽略边缘世界

2. **焦点细采样**
   - 在 `gaze_center` 周围提升采样密度
   - 支持“盯着看就看得更细”

3. **显著补充采样**
   - 对全局显著图中排名靠前的区域补采样
   - 防止边缘突发事件被错过

### 11.4 视觉采样概率建议

建议：

`P_visual = f(center_bias, contrast, edge_density, motion, novelty, task_relevance, topdown_attention, brightness)`

其中：

1. `center_bias`
2. `contrast`
3. `edge_density`
4. `motion`
5. `novelty`
6. `task_relevance`
7. `topdown_attention`
8. `brightness`

亮度只能作为其中一项，不能单独主导。

### 11.5 视觉基础属性

V1 建议先抽取最便宜、最白箱的属性：

1. `brightness`
2. `contrast`
3. `color_bucket`
4. `edge_strength`
5. `motion_delta`（若有视频）
6. `screen_region`

### 11.6 视觉基础单元对象示意

```json
{
  "id": "vu_0001",
  "object_type": "sensor_unit",
  "sub_type": "visual_patch_unit",
  "content": {
    "patch_kind": "grid_patch",
    "raw_ref": "frame_912_patch_03_07"
  },
  "space": {
    "stable_frame_type": "screen_2d",
    "stable_coord": {"x": 0.42, "y": 0.31},
    "focus_relative_coord": {"dx": -0.08, "dy": 0.04}
  },
  "attributes": {
    "brightness": 0.61,
    "contrast": 0.73,
    "edge_strength": 0.52,
    "motion_delta": 0.0
  },

    "er": 0.84,
    "ev": 0.0
  }
}
```

### 11.7 视觉感受器不直接做什么

V1 视觉感受器不直接做：

1. 完整目标检测大模型推理
2. 复杂 3D 世界建图
3. 端到端场景语义理解
4. 长期物体身份追踪

二期前段先做“可白箱的轻量采样与局部单元产出”更重要。

## 12. 听觉感受器 V1

### 12.1 输入范围

听觉感受器 V1 建议支持：

1. 音频文件
2. 麦克风短窗
3. 视频音轨片段

### 12.2 一期听觉的定位边界

V1 可以先做单通道听觉。  
但必须明确：

> 单通道听觉 V1 不提供真正的空间定位能力。

因此：

1. 不应宣称“只靠听觉就知道谁在左边说话”
2. 具身场景下的双通道定位属于后续版本

### 12.3 听觉最小工作单位

不使用原始采样点。  
建议使用：

1. 短时窗
2. 时频块
3. onset 事件
4. pitch 候选
5. 节奏片段

### 12.4 听觉采样概率建议

建议：

`P_audio = f(loudness, onset_strength, spectral_novelty, voicing_confidence, rhythm_change, task_relevance, topdown_attention)`

其中：

1. `loudness`
2. `onset_strength`
3. `spectral_novelty`
4. `voicing_confidence`
5. `rhythm_change`
6. `task_relevance`
7. `topdown_attention`

### 12.5 听觉短时缓冲

听觉 V1 建议内置 ring buffer 或 echoic buffer。

作用不是“重放旧声音”，而是：

1. 聚合多个 micro-tick
2. 防止 tick 间短事件完全丢失
3. 在 tick 边界输出固定预算刺激包

### 12.6 听觉基础属性

V1 建议先抽取：

1. `loudness`
2. `pitch_candidate`
3. `onset_strength`
4. `rhythm_signature`
5. `spectral_centroid`

## 13. 视觉焦点控制器

### 13.1 为什么要单独拆出来

视觉焦点不是一个普通属性。  
它同时影响：

1. 采样概率
2. 视觉预算分布
3. 后续对象分辨率
4. 视觉行动路径

因此建议单独做 `gaze_controller`。

### 13.2 焦点状态

建议维护：

1. `gaze_center`
2. `gaze_velocity`
3. `gaze_target_id`
4. `dwell_ticks`
5. `last_saccade_tick`

### 13.3 焦点行动候选

建议至少支持：

1. `move_gaze_to(x, y)`
2. `micro_scan(dx, dy)`
3. `hold_gaze(duration)`
4. `track_target(target_id)`

### 13.4 焦点行动竞争

同一感受器每个 tick 只执行一个胜出行动。  
驱动力来源建议包括：

1. 本能显著性驱动
2. 注意力回流
3. 任务目标
4. 学习形成的习惯

### 13.5 防抖要求

必须至少有：

1. `saccade_cost`
2. `dwell_bias`
3. `refractory_period`
4. `track_bonus`

否则视焦点会在多个对象间 ping-pong。

## 14. 从感受器到 HDB-V2 的接口

### 14.1 主接口形式

建议统一为：

1. `ingest_visual_frame()`
2. `ingest_audio_window()`
3. `flush_tick_packet()`
4. `get_runtime_snapshot()`
5. `reload_config()`
6. `clear_runtime_state()`

### 14.2 感受器向主闭环提交什么

每个 cognitive tick，感受器只提交：

1. 当前 `sensor_frame`
2. 当前 `stimulus_packet`
3. 当前 `fatigue_summary`
4. 当前 `sampling_summary`

### 14.3 HDB-V2 如何使用

HDB-V2 不关心感受器底层像素或原始波形。  
它只接收：

1. 视觉/听觉基础单元
2. 属性实例
3. 预算与疲劳摘要
4. 时空/时间上下文

再把它们纳入统一 SA 竞争与后续认知主流程。

## 15. 白箱观测与调试要求

感受器 V1 必须从一开始就带以下白箱面板：

1. 当前视觉焦点位置
2. 当前视觉采样 patch 分布
3. 当前视觉显著图摘要
4. 当前听觉时间窗与被采样片段
5. 当前预算占用情况
6. 当前疲劳抑制命中情况
7. 当前属性实例生成记录
8. 焦点行动候选与驱动力竞争结果

否则后续很难判断：

1. 是感受器没把信息送进来
2. 还是 HDB-V2 没学到

## 16. 性能与复杂度约束

### 16.1 复杂度主张

感受器 V1 的任务，是用前端廉价计算换后端主闭环复杂度稳定。  
因此它必须满足：

1. 感受器原始吞吐量可以高
2. 但主闭环读取长度必须稳定

### 16.2 视觉复杂度建议

视觉每 tick 的主要开销来自：

1. 显著图生成
2. patch 采样
3. 基础属性提取
4. 焦点控制

一期建议全部采用轻量方法，避免一开始就引入大模型视觉编码。

### 16.3 听觉复杂度建议

听觉每 tick 的主要开销来自：

1. ring buffer 管理
2. STFT 或等价短时分析
3. 基础属性提取
4. 固定预算筛选

## 19. 预期目标效果与成功图景

这一章是感受器设计最不该省略的一章。  
因为我们不是为了“做出一个视觉前处理脚本”或“做出一个音频窗口切片器”，而是为了让 AP 二期真正出现你之前描述的那些认知图景。

### 19.1 图景一：扫一眼信息少，持续关注信息变多

这是视觉感受器 V1 最核心的预期效果之一。

成功后应呈现：

1. 第一次看到一张图片时，只进入少量全局粗信息和少量焦点细信息
2. 如果系统没有继续关注某一区域，则该区域不会快速细化
3. 如果注意力持续停留在某区域，则焦点细采样预算逐步向该区域倾斜
4. 同一对象在后续 tick 中呈现出越来越多局部单元和属性实例

这意味着你会看到一种非常重要的认知风格：

> 初见是模糊整体，持续盯住才逐步精细。

### 19.2 图景二：视焦点会自然转移，而不是固定死在一个点

成功后应呈现：

1. 默认状态下，焦点停留在当前中心或近期历史焦点附近
2. 某个对象的显著性、任务相关性或高层注意力回流提升后，其位置会获得更高驱动力
3. `move_gaze_to(...)` 胜出后，焦点移动到对应区域
4. 下一 tick 该区域采样密度提升

你应该看到的不是：

1. 焦点永远死锁在中央
2. 焦点毫无理由随机乱跳

而是：

> 焦点在“有原因”时移动，移动后真的带来细节增量。

### 19.3 图景三：两个对象轮流吸引注意时，焦点会切换，但不会疯狂抖动

这正对应你之前描述的那个画面：

1. A 对象更显著或更相关时，焦点看向 A
2. 过一会儿 B 对象获得更高驱动力时，焦点转向 B
3. 切换后会有驻留，而不是每 tick 在 A/B 之间抽搐

成功后的行为表现应是：

1. 焦点会切换
2. 切换后会稳定几轮
3. 切换会带来后续采样密度变化

### 19.4 图景四：重复背景逐步退隐，变化与新异进入前台

成功后应呈现：

1. 稳定背景亮区、持续底噪、长期无变化区域，最初可以进入，但其刺激量会逐步下降
2. 新变化、突发事件、局部运动、强 onset、更高新异度对象更容易抢到预算
3. 任务相关或已被注意的持续对象，不会因为疲劳而完全消失

也就是说：

> 系统既不会被背景一直喂满，也不会把所有持续对象都变成“完全感觉不到”。

### 19.5 图景五：主闭环复杂度相对稳定

这是工程层面的硬成功目标。

无论外界输入是：

1. 一张简单图片
2. 一段复杂视频
3. 一段高变化音频
4. 多模态同时爆量

主闭环在每个 tick 中看到的，都应是固定上限长度的刺激包。  
换句话说：

> 外界复杂度可以波动，但认知主循环的输入规模不能跟着原始吞吐量失控。

### 19.6 图景六：跨模态桥接开始出现，但不过度装聪明

V1 阶段不追求复杂多模态语义理解，但应出现最基础的桥接现象：

1. 某段声音 onset 明显时，视觉候选对象更容易获得焦点竞争优势
2. 持续盯住的视觉区域，其视觉基础单元越来越密
3. 视觉对象和听觉事件可以在同一 tick 合法共存于完整刺激流中

成功图景是：

1. 桥接开始出现
2. 但不靠虚构复杂语义来“装懂”

### 19.7 图景七：白箱可见，而不是只能靠脑补

最终做出来后，操作者应该能够直接看到：

1. 焦点当前在哪里
2. 为什么动过去
3. 采样了哪些 patch / 哪些音频窗
4. 哪些候选因为预算不够被裁掉了
5. 哪些刺激因为 habituation 被压低了
6. 哪些属性实例真的进入了主闭环

只有这样，后面你在做实验、写论文、调参数时，才知道问题到底出在：

1. 感受器前端
2. HDB-V2
3. 状态池
4. 注意力

## 20. 效率优化策略

感受器 V1 不只是要有认知味道，还要努力不拖垮整个系统。

### 20.1 效率目标

感受器 V1 的效率目标不是零成本，而是：

> **用较小、可预测的前端代价，换来主闭环复杂度的稳定。**

因此真正应该追求的是：

1. 感受器原始吞吐可以高
2. 但主闭环接收长度必须稳定
3. 前端每 tick 的代价要线性可控

### 20.2 优先优化什么

感受器 V1 应优先优化：

1. 输入预算控制
2. 采样筛选成本
3. 疲劳状态维护
4. 中间结果复用
5. 白箱可视化开销

而不应一开始优先追求：

1. 大模型级视觉理解
2. 重型听觉深表征
3. 复杂跨模态端到端模型

### 20.3 视觉侧的效率策略

建议：

1. **先网格化，再筛选**  
   先切 patch / tile，而不是全图自由搜索。

2. **显著图要便宜**  
   V1 只用亮度梯度、对比度、边缘、简单帧差等轻量统计。

3. **高分辨率只投向焦点与少量显著补充区**  
   绝不做全图高分辨率展开。

4. **复用上一 tick 中间结果**  
   对低变化画面尽量做局部增量更新。

5. **只保留少量基础属性**  
   brightness、contrast、edge、motion，先别贪多。

### 20.4 听觉侧的效率策略

建议：

1. **固定窗长 + 固定 hop**  
   避免底层分析粒度随场景摇摆。

2. **小规模 STFT 或等价轻量谱特征**

3. **buffer 内先粗筛，再入预算**

4. **属性集合保持克制**
   loudness、pitch 候选、onset、简单 rhythm 足够起步。

5. **持续底噪提前 habituation 掉**

### 20.5 预算分配器必须统一

预算分配器建议单独做共通模块，防止视觉采样、听觉采样、焦点补采样各自偷偷膨胀。

统一流程建议：

1. 候选池汇总
2. 统一打分
3. Top-K 裁剪
4. 生成固定预算刺激包

### 20.6 疲劳状态维护必须接近 O(1)

不建议维护大而全的长窗口精确历史。  
优先使用 EMA 或很小的状态摘要，这样：

1. 每个 fatigue key 成本接近常数
2. 状态规模更可控

### 20.7 焦点控制器也必须有预算

焦点控制器不是无限制搜索器。  
建议每 tick 只评估少量候选动作，候选来源限定为：

1. 当前显著 Top-K
2. 当前注意回流 Top-K
3. 当前 gaze target 邻域

### 20.8 中间结果要可缓存

建议缓存：

1. 最近一轮视觉 patch 属性
2. 最近一轮显著图摘要
3. 最近一轮音频窗特征
4. 最近一轮预算裁剪结果

供：

1. 下一 tick 增量复用
2. 观测台直接展示

### 20.9 预期效率成功标准

如果感受器设计得好，最终应看到：

1. 原始输入吞吐量上升时，主闭环耗时增长明显更慢
2. 前端感受器的额外成本大体可预测
3. 大多数 tick 的延迟分布更稳定
4. 不会出现单帧或单段音频把主系统直接拖死

## 21. 验收标准

### 19.1 功能验收

1. 能从图片/截图产生视觉刺激包
2. 能从音频短窗产生听觉刺激包
3. 每个 tick 输出长度稳定
4. 可输出属性实例
5. 可输出统一 `sensor_frame` 与 `stimulus_packet`

### 19.2 认知风格验收

1. 焦点附近采样明显更密
2. 持续关注同一区域时，细节量逐步提升
3. 重复背景逐步降噪
4. 突发变化更容易进入预算

### 19.3 白箱验收

1. 当前采样区域可展示
2. 当前疲劳抑制项可展示
3. 当前预算裁剪理由可展示
4. 焦点行动竞争可展示

### 19.4 稳定性验收

1. 主闭环输入长度稳定
2. 焦点移动不严重 ping-pong
3. 音频不会把同一证据重复当作全新外源刺激

## 23. 风险清单

### 风险 1：感受器前端变成黑箱

规避：

- 每 tick 保留采样摘要
- 采样理由可展示
- 预算裁剪理由可展示

### 风险 2：预算固定后把重要信息饿死

规避：

- 全局粗采样保底
- 显著补采样
- 任务相关增益

### 风险 3：重复疲劳过强导致持续重要对象消失

规避：

- fatigue 与 task/attention gain 联动
- 设最小保留增益

### 风险 4：焦点中心与稳定坐标混用

规避：

- 稳定坐标与焦点相对坐标分开保存

### 风险 5：连续流与认知 tick 混淆

规避：

- 明确 micro-tick 与 cognitive tick 解耦

## 24. 示例流程

### 示例 1：视觉扫一眼与持续盯视

1. 图片进入视觉感受器
2. 全局粗采样产出少量 patch
3. 焦点位于中央，因此中央区域多拿到预算
4. 若下一 tick 注意力仍停留在同区域，则焦点细采样继续增密
5. 系统对同一对象的认知粒度逐步变细

### 示例 2：底噪逐步淡出

1. 空调噪声连续多个 tick 都存在
2. 听觉疲劳状态逐步上升
3. 其进入主闭环的刺激量逐步下降
4. 但若当前任务恰好在监听该噪声异常变化，则 task gain 可把它重新拉回

### 示例 3：说话触发看向人脸

1. 听觉感受器检测到强 onset 与较高 voicing
2. 视觉感受器已有多个候选人脸区域
3. 上层竞争让某候选人脸获得更高焦点驱动力
4. `move_gaze_to(face_x, face_y)` 获胜
5. 下一 tick 该区域采样密度提高

### 示例 4：两个对象轮流吸引注意，焦点自然切换

假设画面中有对象 A 和对象 B。

1. 初始时 A 的显著性更高
2. 焦点行动竞争中，`move_gaze_to(A)` 获胜
3. 焦点移动后，A 周围获得更高细采样预算
4. 几轮后 B 突然发生更大的变化，或高层任务把 B 的优先级拉高
5. 新一轮焦点竞争中，`move_gaze_to(B)` 获胜
6. 焦点平滑切换到 B

理想结果是：

1. 焦点确实会切换
2. 切换后有短时驻留
3. 焦点不会每 tick 在 A/B 间来回抽搐

### 示例 5：外界输入很大，但主闭环刺激包长度仍然稳定

假设当前同时发生：

1. 一张复杂截图进入
2. 一段音频短窗进入
3. 一条文本输入进入

即便原始输入量很大，主闭环仍然只应读到：

1. 固定上限的视觉刺激单元
2. 固定上限的听觉刺激单元
3. 固定上限的文本刺激单元

这说明：

1. 感受器前端已经完成预算压缩
2. 主闭环复杂度没有随着原始流爆炸
3. 系统整体延迟仍然可控

## 25. 最终结论

感受器 V1 的本质不是“做出完整视觉/听觉智能”，而是：

> **在不压垮 AP 主闭环的前提下，把高吞吐连续多模态输入压缩成固定预算、白箱可观察、可进入 HDB-V2 的刺激对象集合。**

如果这一步设计正确，二期后面的：

1. HDB-V2
2. 状态池
3. 注意力
4. 视焦点行动
5. 长期多模态学习

才会有一个真正稳的输入底座。
