# AP 二期 Phase17B 观测台编辑器性能与 Runtime Digest 补强验收记录

日期：2026-05-19

## 本轮目标

这一轮不改 AP 二期的主链哲学，不改 HDB-V2、状态池、记忆召回、规则引擎的核心口径，重点只做两件事：

1. 把规则编辑器 / 调参编辑器在“大量规则或档位”下的工作流继续做轻
2. 把 `Runtime 结构` 页签再往“研究控制台”方向上翻一层，减少纯表格和 JSON 依赖

## 本轮新增改进

### 1. 编辑器草稿同步机制升级

规则编辑器和调参编辑器不再只是“每改一个输入框就整表扫描”。

本轮新增：

1. 单卡同步
   - 输入发生时优先更新当前卡片对应的草稿对象
2. 去抖汇总
   - 延迟少量时间后，再统一汇总整张编辑表
3. 状态线显式显示：
   - `表单已同步`
   - `输入待同步`
4. 待同步计数
   - 能直接看到最近连续输入积累了多少次待同步变更

这意味着：

1. 长时间编辑时，状态反馈更可信
2. 大列表输入时，前端重活更少
3. “我刚刚改的东西到底有没有进草稿”这件事更清楚

### 2. 编辑器筛选链减负

本轮把这些操作优先改成“显隐更新”，而不是默认整表重建：

1. 规则区：
   - `ruleSearch`
   - `family`
   - `enabled`
   - `audit`
2. 调参区：
   - `tunerSearch`
   - `target`
   - `enabled`
   - `audit`
3. warning 过滤切换：
   - level
   - code
   - path/text

这一步的目标非常明确：

1. 大规模规则集下搜索更顺
2. 调规则时切过滤条件不会整页喘
3. 保留现有审计 / warning / 定位能力，但减少不必要重绘

### 3. 编辑器新增快捷骨架

为了减少“每次都从空条件和空效果慢慢填”的摩擦，本轮给规则和调参都加了快捷插入按钮。

#### 条件快捷插入

1. `高耗时保护`
2. `锚点主线`
3. `已有一级召回`
4. `期待占优`
5. `违和感抬升`

#### 效果快捷插入

1. `违和感地板`
2. `注入违和感 SA`
3. `继续聚焦驱动`
4. `记录规则日志`

#### 调参快捷插入

1. `聚焦增益`
2. `采样预算`
3. `后继优势`
4. `锚点偏置`
5. `违和感增益`

这些骨架不是替代自定义编辑，而是把最常见的研究动作提到前台，减少手工字段填充成本。

### 4. Runtime Digest 卡

`Runtime 结构` 页签新增了一层 digest 卡，而不是要求操作者先读表格或进 JSON。

新增摘要：

1. `状态池摘要`
2. `向量 / Posting / 时空`
3. `短期记忆主线`
4. `执行器摘要`

其目的不是替代表格，而是给操作者一个“第一眼扫视层”：

1. 先看系统现在大体在什么状态
2. 再决定要不要进入下方细表
3. 再决定要不要进入 JSON 深挖

### 5. 编辑器同步状态统一复位

以下入口现在会统一清理“待同步”状态：

1. 刷新规则 / 调参
2. 重置规则 / 调参
3. 导入规则 / 调参 JSON
4. 保存规则 / 调参

这样可以避免：

1. 状态条残留旧的待同步计数
2. 已经保存/导入/重置后，前端还像没同步完一样

## HTTP 与真页验证

### 1. HTTP 返回源码校验

已通过 HTTP 拉取 `http://127.0.0.1:8766/` 返回页面源码，确认存在：

1. `runtimePoolDigestBox`
2. `runtimeIndexDigestBox`
3. `runtimeShortTermDigestBox`
4. `runtimeExecutorDigestBox`
5. `data-role="quick-add-condition"`
6. `data-role="quick-add-effect"`
7. `data-role="quick-add-adjustment"`

### 2. 浏览器真页验证

使用新开标签页访问带时间戳的 URL 后，确认：

1. 页面标题为：
   - `AP 二期观测台 V2`
2. `Runtime 结构` 页签切换后，真实可见：
   - `状态池摘要`
   - `执行器摘要`
   - digest 内容文本
3. 编辑器模板区与快捷骨架按钮在真页 DOM 中存在

### 3. 一个已确认的环境边界

本轮再次确认：

1. HTTP 返回的 HTML 可能已经是新版本
2. 但旧浏览器标签页仍可能残留旧 DOM

因此更稳的真页验收方式仍然是：

1. 先拉 HTTP 页面源码
2. 再新开标签页访问带时间戳的 URL
3. 不只依赖旧页签 `reload()`

## 本轮意义

这一轮的价值不在于“又多了几个控件”，而在于把观测台继续往专业研究控制台方向推进了一层：

1. 编辑器在大规模规则 / 档位下更能长时间工作
2. 搜索和筛选的代价更低
3. 规则 / 调参常用骨架更容易快速试验
4. runtime 运行态更容易第一眼看懂
5. 前端状态条对“是否已进入草稿”的表达更可信

## 仍可继续补强的方向

1. 若规则数量进一步扩大到数百上千条，可以继续考虑：
   - 可见区虚拟滚动
   - 折叠卡保留 DOM、少重建
2. `Runtime 结构` digest 卡之后，还可以继续给：
   - `Sidecar 结构`
   - `Live 结构`
   增加同类 digest 层
3. 快捷骨架后续可以继续接“按 family / target 推荐模板”的更强工作流
