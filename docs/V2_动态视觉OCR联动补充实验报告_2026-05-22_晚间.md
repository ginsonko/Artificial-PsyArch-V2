# V2 动态视觉 OCR 联动补充实验报告

生成时间：2026-05-22 晚间

## 1. 目标

本轮不是重复证明“动态视觉对象层存在”，而是进一步回答一个更严格的问题：

1. 动态视觉链条在 OCR-like 场景里，是否能让识别更早或更稳？
2. 之前观测到的提升，究竟来自真实的动态联动，还是实验污染？
3. 如果结果不理想，底层真正的瓶颈在哪里？

## 2. 本轮先修掉的实验污染

### 2.1 自动惊回看污染

此前 `execute_selected_actions=False` 时，运行链里仍可能触发自动惊回看，导致所谓“被动观察”并不纯粹。

已正式加入开关：

- `vision_auto_surprise_reorient_enabled`

并通过测试验证可关闭。

### 2.2 probe 边测边写长期记忆

动态 OCR probe 在观测阶段原本仍会写入：

- `focus_chain`
- `exact_external`
- `latent_state_snapshot`

这会让“连续看清楚”与“测试时又学了一遍”混在一起。

本轮在 probe 脚本中加入了只读模式：

- `--probe-read-only`

只读模式下，probe 观测阶段不再写入长期记忆。

### 2.3 移动后视觉身份被位置绑死

之前 `vision_mem::...` 的特征标签里，结构信息和位置桶信息直接揉在一个身份标签里。
这意味着：

- 同一个字一旦移动，
- 即使结构仍然相似，
- 检索层也更容易把它当成“另一个对象”。

本轮修复方式不是删掉空间信息，而是增加“结构检索别名”：

- `vision_core::...`
- `vision_form::...`
- `vision_global::...`

这些别名保留结构共性，但不把当前位置硬绑进身份本体。
位置仍然继续通过：

- `coords`
- `spacetime`
- 查询向量中的空间 token

参与匹配。

这更符合“结构是结构，位置是位置”的设计哲学。

## 4.1 实验 A：`digit_3` 低预算动态联动 probe（未只读）

输出目录：

- `outputs/dynamic_ocr_coupling/20260522_202550_discrim_d3_lowbudget/`

设置：

- 训练：`4` epochs（已通过训练验证）
- probe：`raw=32 / patch=8 / focus=4`
- 画布：`512 x 192`
- 位移：`0,40,80,120,160,200`
- 条件：
  - `moving_passive_no_dynamic`
  - `moving_dynamic_auto`
  - `moving_dynamic_full`

结果摘要：

- passive：`success_count=2`
- dynamic_auto：`success_count=4`
- dynamic_full：`success_count=3`

### 结论

这组结果**不能直接当作动态联动有效的证据**。

原因：

1. probe 当时仍在写长期记忆
2. `Bn` 顶部出现了 probe 阶段新写入的空文本 `exact_external`
3. 这会把真实的“连续识别”与“测试期间再次学习”混在一起

因此这组结果只能作为“发现污染信号”的线索，不能作为正式证据。

## 4.2 实验 B：`digit_3` 低预算动态联动 probe（只读）

输出目录：

- `outputs/dynamic_ocr_coupling/20260522_203020_discrim_d3_lowbudget_readonly/`

设置与实验 A 基本一致，但增加：

- `--probe-read-only`

结果摘要：

- `moving_passive_no_dynamic`
  - `first_success_tick = 0`
  - `success_count = 5`

- `moving_dynamic_auto`
  - `first_success_tick = 0`
  - `success_count = 5`

- `moving_dynamic_full`
  - `first_success_tick = 0`
  - `success_count = 5`

### 结论

在只读条件下，`digit_3` 这个任务本身已经过于容易：

- 被动模式就足以维持稳定识别
- 动态联动没有展示出更早的成功时刻
- 因而这组实验**不具备判别力**

它说明：

1. 我们之前观测到的“动态增益”里，确实混有边测边写的成分
2. 去掉污染后，`digit_3` 低预算任务本身太容易，不足以证明动态联动的附加价值

## 4.3 实验 C：`digit_8` 超低预算动态联动 probe（只读）

输出目录：

- `outputs/dynamic_ocr_coupling/20260522_203154_discrim_d8_ultralow_readonly/`

设置：

- 训练：`4` epochs（已通过训练验证）
- probe：`raw=16 / patch=4 / focus=2`
- 画布：`640 x 224`
- 位移：`0,80,160,240,320`
- 条件：
  - `moving_passive_no_dynamic`
  - `moving_dynamic_auto`
  - `moving_dynamic_full`
- probe：只读

结果摘要：

### `moving_passive_no_dynamic`

- `first_success_tick = 2`
- `success_count = 3`

逐 tick：

- tick0：`Bn` 空，`C* = eight`
- tick1：`Bn` 空，`C* = eight`
- tick2~4：严格成功

### `moving_dynamic_auto`

- `first_success_tick = 2`
- `success_count = 1`

逐 tick：

- tick0：`Bn` 空，`C* = eight`
- tick1：`Bn` 空，`C* = eight`
- tick2：成功
- tick3~4：再次掉回 `Bn` 空

### `moving_dynamic_full`

- `first_success_tick = 2`
- `success_count = 2`

逐 tick：

- tick0：`Bn` 空，`C* = eight`
- tick1：`Bn` 空，`C* = eight`
- tick2：成功
- tick3：失败
- tick4：成功

### 结论

这组实验已经具有真正的判别力，而且结果很清楚：

1. 动态联动**没有**把首次成功时刻提前
2. 动态联动在当前参数下**没有**比被动模式更稳
3. 在这个任务上，被动模式反而更稳定

因此，到本轮结束为止：

- “动态视觉联动能显著提升 OCR-like 识别稳定性”
  - **尚未被证明**

## 5. 根因分析

### 5.1 已排除的错误解释

不是因为：

1. 动态对象层根本没工作
   - 它已经工作，并能产出动态对象摘要

2. 自动惊回看没触发
   - 它已经触发，并显著抬高了采样预算

3. 之前结果不好只是位置变化导致彻底失配
   - 本轮通过结构检索别名已经缓解了这点
   - 至少检索层现在不再被位置硬绑死

### 5.2 当前最核心的瓶颈

从只读 probe 的 `recall_query_preview` 与 `bn_head` 看，当前真正的问题更像是：

#### 问题 A：动态对象摘要和全局结构特征过强

动态联动开启后，query 顶部更容易被以下项占满：

- `vision_dyn::trk_*`
- `vision_mem::global_*`

这些信息适合：

- 发现“有东西在动”
- 把注意力拉过去

但它们不够适合：

- 区分“这是 3 还是 8”

也就是说：

- 动态摘要更像“注意力控制信息”
- 而不是“身份识别信息”

#### 问题 B：局部结构样本没有随着 boost 同步占据 query 主导

虽然动态模式下 raw budget 从：

- `16 -> 79/113/208`

确实被拉高了，但 recall query 的顶部仍然经常被：

- 全局结构
- 动态轨迹
- 旧预测文本

占据。

这意味着：

- “看得更多”并没有自动变成“更有区分力地看”

#### 问题 C：`Bn` 严格成功仍容易卡在 exact branch 排序上

在不少失败 tick 中：

- `C*` 和 `state` 其实已经是正确的
- 但 `Bn` 侧 exact 候选没有顶到最前

这说明当前链路更接近：

- “整体预测已大致认出来”

但还没有稳定达到：

- “精确回忆项也总能在第一位被拉出来”

## 6. 当前可以严谨声称的结论

### 6.1 已被证明

1. 动态视觉对象层本身成立
2. 自动惊回看与采样预算提升成立
3. 视觉结构检索别名方案是可行的，且未破坏现有链路
4. probe 已被净化到更接近真实评估

### 6.2 尚未被证明

1. 动态视觉联动能让 OCR-like 识别更早发生
2. 动态视觉联动能让 OCR-like 识别比被动模式更稳

### 6.3 本轮新增的重要负结论

在更干净、更严的只读实验里：

- 之前看上去的部分动态增益，确实包含实验污染成分

这不是坏事，反而提高了我们后续实验的可信度。

## 7. 下一步最值得继续做的事

### 7.1 不建议继续硬堆 raw budget

因为当前问题已经不是“看得不够多”，而是：

- query 里什么信息占主导

### 7.2 更合适的下一步

建议优先做两类软修正：

1. 让 `vision_dyn::*` 更偏向注意力控制，而不是直接强占 recall query 主导
2. 在有新鲜局部结构样本时，提高 `visual_focus_feature_unit` 对 recall query 的占比

注意：

- 这不是数字专属规则
- 也不是硬编码 3/8
- 而是区分“运动/对象性信息”和“身份结构信息”的系统角色

## 8. 总结

本轮最重要的价值，不是把动态 OCR 联动“做成了”，而是把这个问题从混浊状态拉到了清晰状态：

1. 我们修掉了实验污染
2. 我们修掉了视觉结构被位置硬绑定的问题
3. 我们证明了当前动态联动并未自然带来 OCR-like 增益
4. 我们把剩余瓶颈收敛到了“query 主导信息类型不对”

所以当前最严谨的判断是：

- 动态视觉底层链条是通的
- 但“动态联动提升 OCR-like 识别”这一条，目前还不能宣称已经成立
- 下一步应该继续优化“运动信息如何服务识别，而不是盖过识别”
