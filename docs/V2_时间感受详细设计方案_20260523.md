# AP 二期 V2 时间感受详细设计方案

日期：2026-05-23

## 1. 文档目的

本方案用于把“时间感受”从当前二期已经存在的客观时空索引之上，正式补成主体可认知、可召回、可反向调制后续匹配的一类统一通道感受。

它遵守《V2 统一通道与感受哲学冷保存》中的总规则。

本方案不把固定桶作为认知定义，而只允许底层在索引阶段使用时间桶缩小候选集。

---

## 2. 当前已有底座

当前二期已具备：

1. 每条记忆都带 `tick_index`
2. `MemoryStoreV2._infer_spacetime(...)` 会写入 `spacetime.t`
3. `SpacetimeIndexV2` 已支持时间近邻候选检索
4. `recall_fatigue` 已能抑制同一记忆被连续重复召回
5. `commitment` 已能评估当前预测对象的主导性

因此当前缺失的不是“时间坐标”，而是：

1. 从召回和预测中抽取主观时间间隔感
2. 把该感受变成 SA
3. 让它反向调制时间相关召回

---

## 3. 时间感受的正式定义

时间感受不是当前时钟本身，而是：

> 当前主体围绕某一批高承诺度记忆，形成的“距离现在多久”的主观时间间隔结构。

也就是说：

1. 记忆中保存的是客观发生时刻 `t_memory`
2. 当前 tick 为 `t_now`
3. 时间间隔为 `delta_t = t_now - t_memory`
4. 当一批高承诺度记忆的 `delta_t` 在某个范围内明显聚集时，主体形成时间感受

这个时间感受本质上是“时间差的主观峰”，而不是某个单记忆的附属字段。

---

## 4. 时间感受生成流程

## 4.1 输入来源

时间感受不直接来自原始外界输入，而主要来自：

1. BN 中高分显式记忆
2. C* 中高承诺度预测对象的支持源
3. 必要时也可参考 latent state snapshot 的时刻分布

但正式第一版优先建议：

- 仅从 BN 显式记忆对象开始
- 这样更稳定、解释性更强

---

## 4.2 候选时间差计算

对每个参与聚合的候选记忆 `i`，计算：

```text
delta_t_i = t_now - t_i
```

并为该记忆赋予一个聚合权重：

```text
w_i
= recall_score_i
 * commitment_i
 * reality_weight_i
 * fatigue_multiplier_i
```

其中：

1. `recall_score_i`
   - 当前 BN 召回得分

2. `commitment_i`
   - 当前该记忆在整体判断中的主导性

3. `reality_weight_i`
   - 真实经历权重

4. `fatigue_multiplier_i`
   - 用于防止同一时间段记忆连续垄断

---

## 4.3 连续时间峰聚合

这里不采用固定桶作为认知定义。

正式建议采用连续峰聚合：

### 方法 A：核密度聚合

对每个 `delta_t_i` 在时间轴上投一个核：

```text
K_i(d) = w_i * exp(-(d - delta_t_i)^2 / (2 * sigma_i^2))
```

总峰为：

```text
TimeField(d) = sum_i K_i(d)
```

然后找 `TimeField` 的主峰。

### 方法 B：工程简化版

在工程第一版中，可以不真的扫整条连续轴，而是：

1. 先按时间近邻取若干候选
2. 用每个候选的 `delta_t_i` 作为种子中心
3. 统计其附近半径 `r_i` 内的加权质量
4. 选主峰

形式类似：

```text
cluster_mass(i)
= sum_j w_j * max(0, 1 - abs(delta_t_j - delta_t_i) / r_i)
```

这仍是连续距离函数，不是固定桶判定。

正式建议先上方法 B。

---

## 4.4 时间感受对象的标准结构

当主峰满足条件时，生成一个主观时间感受 SA：

```json
{
  "sa_label": "timefelt::elapsed",
  "display_text": "时间间隔感",
  "channel": "time",
  "sa_kind": "temporal_feeling_unit",
  "energy": 0.46,
  "attributes": {
    "delta_t_norm": 0.18,
    "delta_sigma_norm": 0.06,
    "confidence": 0.71,
    "cluster_mass": 0.64,
    "dominance": 0.42,
    "source_count": 3
  }
}
```

解释：

1. `delta_t_norm`
   - 归一化后的主观时间差中心

2. `delta_sigma_norm`
   - 模糊范围

3. `confidence`
   - 峰是否清晰稳定

4. `cluster_mass`
   - 该峰聚合到的总质量

5. `dominance`
   - 第一峰相对第二峰的优势度

6. `source_count`
   - 支持该时间感的候选记忆数

正式上，时间感受对象不应直接叫“昨天桶 3”之类名字。

---

## 4.5 时间感受的注入条件

建议采用统一形式：

```text
signal_strength = cluster_mass * dominance
confidence = peak_confidence
felt_energy = gain * max(0, signal_strength - threshold) * confidence
```

推荐第一版参数方向：

1. `time_feeling_threshold = 0.18 ~ 0.25`
2. `time_feeling_gain = 0.8 ~ 1.2`
3. `time_feeling_min_confidence = 0.22`

只有同时满足：

1. 主峰显著
2. 置信度足够
3. 支持源不是单点脆弱噪声

才注入状态池。

---

## 5. 时间感受反向调制召回

## 5.1 基本原则

当状态池中存在显著的时间感受 SA 时，它应反过来增强时间上匹配的记忆召回。

但这种增强必须是软约束，不是硬过滤。

---

## 5.2 调制方式

把当前时间感受变成一个查询意图：

```text
target_delta_t = timefelt.delta_t
target_sigma_t = timefelt.sigma_t
target_t = t_now - target_delta_t
```

对任意记忆 `m`：

```text
memory_delta_t = t_now - t_m
interval_match(m)
= exp(-(memory_delta_t - target_delta_t)^2 / (2 * target_sigma_t^2))
```

最终召回分数：

```text
score_final
= base_score
 + gamma * felt_energy * interval_match
```

其中：

- `gamma` 为时间感受调制增益
- `felt_energy` 为该时间感在状态池中的当前能量

正式建议：

- 不直接改写向量
- 而是在候选重排阶段作为额外连续项加入

---

## 5.3 与现有 `query_spacetime` 的关系

当前 `query_spacetime` 里已经有空间相关量。

建议扩展为：

```json
{
  "t": 120,
  "target_t": 108,
  "target_delta_t": 12,
  "time_sigma": 3.5,
  "time_confidence": 0.68
}
```

然后：

1. `SpacetimeIndexV2` 仍可先用时间桶做候选过滤
2. 真正评分时使用连续 `time_match`

也就是说：

- bucket 仅用于先找附近候选
- 最终分数仍由连续距离决定

---

## 6. 时间感受如何参与语言与事件回忆

以“昨天吃了什么”为例：

1. 文本中的“昨天”通过以往记忆共现，能召回某类时间感受 SA
2. 该 SA 形成 `target_delta_t`
3. 时间匹配会把候选记忆拉向那一段历史
4. “吃”相关语义再在该时间带内筛选
5. 最终回忆出更符合时间条件的吃饭事件

因此时间感受不是替代语义，而是给语义回忆加一条时间轴上的软聚焦。

---

## 7. 与节奏感的关系

节奏感可视为时间感受的高级模式：

1. 时间感受关注的是“离现在多久”
2. 节奏感关注的是“某类事件是否按近似固定时间差重复出现”

所以节奏感不应另起哲学体系，而应建立在时间感受之上。

正式路径：

1. 先实现单次时间间隔感
2. 再实现重复时间差模式的稳定检测
3. 最后再让该模式生成节奏期待感、节奏把握感

---

## 8. 防止时间感受无限循环

时间感受非常容易出现“总在想刚才那件事”的自激。

正式限制建议：

1. 利用现有 `recall_fatigue`
2. 对同一主时间峰增加 `temporal_feeling_fatigue`
3. 时间感受对自身通道的反向调制应有上限

例如：

```text
effective_time_gain
= base_gain * (1 - temporal_feeling_fatigue)
```

并且：

- 每次同一时间峰成为主峰时，疲劳上升
- 若近期未再次被激活，则疲劳快速衰减

这样短期可以“回味”
但不会长期一直锁死在同一个时间感上。

---

## 9. 第一版工程落地点

推荐最小落地顺序：

1. `memory/memory_store_v2.py`
   - 新增 `build_temporal_feeling(...)`
   - 输入 BN 行、当前 tick、候选支持
   - 输出主时间感结构

2. `core/runtime_v2.py`
   - 在 BN / C* 之后调用时间感构建
   - 若满足阈值，则把 `timefelt::elapsed` 注入状态池
   - 同时构造扩展后的 `query_spacetime`

3. `memory/spacetime_index_v2.py`
   - 保留时间桶索引
   - 新增连续 `time_match` 打分入口

4. `config/runtime_config.json`
   - 新增时间感受参数

5. `tests`
   - 增加时间感受生成、时间反向召回、时间疲劳恢复测试

## 11. 验收实验建议

### 11.1 时间感生成实验

目标：

- 当前输入触发某些过去事件的召回时
- 系统是否能稳定生成主时间感

验收：

- 是否出现 `timefelt::elapsed`
- 其 `delta_t_norm / confidence / source_count` 是否合理

### 11.2 时间反向召回实验

目标：

- 加入时间感后
- 是否能把候选召回重排到更接近目标时间差的记忆

验收：

- top1 / top3 是否比无时间感时更接近目标时段

### 11.3 防自激实验

目标：

- 时间感是否会无限霸占回忆

验收：

- 连续多 tick 后是否疲劳上升
- 停止相关刺激后是否逐步恢复

### 11.4 语言问答实验

目标：

- 在“昨天 / 刚才 / 前面 / 上一轮”这类时间线索下
- 是否更容易召回对应事件

---

## 12. 本方案的正式地位

从本方案开始，后续时间感受、节奏感、事件时间线回忆等设计，都默认以本方案为基线，不再把固定桶作为认知定义本体。

如果未来要改动，应在新的冷保存文档中显式覆盖，而不是在实现中偷偷变形成“按桶思考”的旧路线。
