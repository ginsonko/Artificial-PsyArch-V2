# AP 二期视频流与自主循环入口说明

## 一、真视频文件入口

当前已经支持：

- CLI：`run-video-stream`
- Web API：`POST /api/runs/video-stream/start`
- BAT：[启动视频流样例实验.bat](../启动视频流样例实验.bat)

### 设计口径

这次不是重写一套新的视觉主链，而是：

1. 用 OpenCV 解码视频文件
2. 按 `stride` 或 `tick_fps` 抽帧
3. 每一帧转成 PNG bytes
4. 复用现有 `VisionSensorV1`
5. 继续走原有 tick 主链

也就是说，它仍然符合这期一直坚持的原则：

- 主链统一
- 低风险扩展
- 不平行再造第二套系统

### CLI 用法

```bash
python -m observatory_v2 run-video-stream --video demo.mp4 --tick-fps 4 --text-prefix "观察视频"
```

可选参数：

- `--tick-fps`
- `--frame-stride`
- `--max-frames`
- `--interval-ms`
- `--reset-runtime`

### Web API

接口：

- `POST /api/runs/video-stream/start`

字段：

- `video_b64`
- `text_prefix`
- `tick_fps`
- `frame_stride`
- `max_frames`
- `tick_interval_ms`
- `reset_runtime`

## 二、自主循环入口

当前已经支持：

- CLI：`run-autonomous`
- Web API：`POST /api/runs/autonomous/start`
- BAT：[启动自主循环样例.bat](../启动自主循环样例.bat)

### 现在的主链

每个自主 tick 会走：

1. 自动截图
2. 图像进入视觉感受器
3. 进入统一 runtime 主链
4. rules 产出动作驱动力
5. sandbox / executor 选择动作
6. 动作效果回写
7. 动作反馈进入 action learning / tuner learning
8. 结果写入 summary / sidecar / metrics / rollup

### 当前新增的长期运行保护

已经加入：

- 连续截图失败阈值
- 连续动作错误阈值
- 连续空转阈值
- idle backoff
- 自动反馈塑形

## 三、教师层与 LLM 门控的定位

这次新增的教师层与 LLM 门控，必须按下面的哲学理解：

1. **AP 先产生主链**
   - 视觉/听觉/文本先进入 AP 主链
   - `Bn / C_i / C* / rules / action_drives` 先由 AP 自己算出来

2. **教师层只做稀疏监督**
   - 审阅动作候选
   - 对明显高风险、低把握动作进行阻断或降权
   - 根据结果给予稀疏奖励/惩罚
   - 记录解释与审计信息

3. **LLM 不是主脑**
   - 当前工程里，LLM 门控默认仍以 `heuristic` 为主，外部教师已支持 `stub_file / http_json`
   - 以后即使接真实 LLM，也只能做：
     - 辅导
     - 审阅
     - 安全 guardrail
     - 稀疏教师信号
   - 不能替代 AP 自己的预测、注意力竞争和长期塑形

4. **教师信号会回到 AP 内部**
   - 奖励信号
   - 惩罚信号
   - 教师审阅结果

这些不会只停在日志里，而会进入：

- `action_learning`
- `tuner_learning`
- 状态池注入
- 长期记忆写入

这样最终要证明的是：

- AP 本身具有学习能力
- LLM 只是初期教师和安全层
- 后期可以逐步退化为观察员、解释器、安全员

### 自动反馈逻辑

如果开启 `auto_feedback_enabled`，系统会根据：

- 截图是否成功
- 是否真的执行了动作
- 是否有 gaze / cursor 变化
- 是否连续空转
- 是否连续动作错误

自动生成一小份：

- `reward`
- `punishment`
- `notes`

再和用户显式传入的 `external_feedback` 合并。

这意味着：

- 即使没有人工逐 tick 奖惩
- 系统也已经有了一个弱监督的长期塑形通道

### CLI 用法

```bash
python -m observatory_v2 run-autonomous ^
  --ticks 12 ^
  --text-hint "观察当前桌面并持续聚焦" ^
  --stop-on-capture-failures 3 ^
  --stop-on-action-errors 4 ^
  --stop-on-idle-ticks 6 ^
  --idle-backoff-ms 150
```

新增关键参数：

- `--stop-on-capture-failures`
- `--stop-on-action-errors`
- `--stop-on-idle-ticks`
- `--idle-backoff-ms`
- `--auto-feedback`

其中 `--auto-feedback` 支持：

- `default`
- `on`
- `off`

## 四、可观测性

这两类运行现在都能在日志里重建出：

- 每 tick 输入
- 视频帧采样信息
- 自主 tick 保护阈值
- 自动反馈结果
- 运行停止原因

尤其自主循环会额外暴露：

- `autonomous_summary`
- `autonomous_sidecar`

便于后续长期研究和审计。

## 五、2026-05-19 新增底层能力补充

### 1. 外部教师协议已升级

当前外部教师不再只有 `stub_file`，而是已经支持正式 provider 模式：

- `off`
- `stub_file`
- `http_json`

并且已经有正式 request / response schema：

- `external_teacher_request/v1`
- `external_teacher_response/v1`

这意味着：

- 现在可以继续保持 AP 主链自己认知、自己产出动作候选
- 外部教师只做审阅、奖惩、警告、阻断
- 后续如果接真实联网教师 provider，不需要再推翻现有教师层

### 2. 持续 autonomous session 已可用

除了原本的 batch 式：

- `run-autonomous`
- `POST /api/runs/autonomous/start`

现在还新增了持续 session 形态：

CLI：

- `run-autonomous-session`
- `pause-autonomous-session`
- `resume-autonomous-session`
- `stop-autonomous-session`
- `autonomous-session-status`

Web API：

- `GET /api/autonomous-session/status`
- `POST /api/autonomous-session/start`
- `POST /api/autonomous-session/pause`
- `POST /api/autonomous-session/resume`
- `POST /api/autonomous-session/stop`

### 3. realtime source abstraction 已统一

当前 audio / image / video / screen 已经开始走统一 source 抽象。

统一 source 已支持：

- `next_item()`
- `status()`
- `close()`

当前已接入：

- `audio_file`
- `image_sequence`
- `video_file`
- `screen_capture`
- `webcam`
- `microphone`

其中：

- `audio_file / video_file` 已升级为按 tick 懒加载消费，不再默认先整条流全部展开到内存
- `webcam / microphone` 不再只是占位接口，而是已经具备真实设备采集适配器
- 如果当前机器没有对应设备、驱动或依赖，也会明确走 `unavailable` / `AppError` 路径，而不是静默崩掉

### 4. webcam / microphone 入口

CLI：

- `run-webcam-stream`
- `run-microphone-stream`

Web API：

- `POST /api/runs/webcam-stream/start`
- `POST /api/runs/microphone-stream/start`

说明：

1. `webcam` 通过 OpenCV `VideoCapture` 抓帧
2. `microphone` 通过 `sounddevice` 按固定窗口录音
3. 两者都会统一包装成 `realtime source`
4. 每个 tick 只拉取当前需要的一帧或一个音频窗，避免长流输入一次性拉爆内存

安装建议：

- `opencv-python>=4.8`
- `sounddevice>=0.4.7`

如果只是跑文本、图片、音频文件、视频文件样例，不强制需要真实设备流依赖。
